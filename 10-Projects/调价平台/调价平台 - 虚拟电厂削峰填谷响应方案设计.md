# 调价平台 - 虚拟电厂削峰填谷响应方案设计

## 目标

梳理调价平台在虚拟电厂调控场景下的业务定位、逻辑关系和方案边界，明确车网互动大脑接收削峰、填谷指令后，调价平台如何通过价格和优惠活动影响用户充电行为，并形成可上报、可执行、可评估的响应策略。

## 背景

车网互动大脑会接收虚拟电厂下发的调控指令。调控指令通常用于让充电负荷参与电网侧调节，核心类型分为两类：

- **削峰**：电网或虚拟电厂希望在高负荷时段降低充电负荷，避免峰值继续抬升。
- **填谷**：电网或虚拟电厂希望在低负荷时段提升充电负荷，提高低谷电能消纳能力。

当前要做的是调价平台。调价平台不是直接控制充电枪功率的执行系统，而是面向运营侧，通过价格、优惠活动、站点推荐和渠道分发影响用户选择，从而间接改变站点负荷曲线。

因此，整体业务可以拆成两条线：

- **执行侧**：车网互动大脑根据虚拟电厂指令，对场站、设备或充电功率进行调控，尤其适用于削峰中的限功率、降额、轮控等动作。
- **运营侧**：调价平台根据调控目标、站点历史响应情况和收益测算，生成价格或优惠策略，并同步到甬易充平台、充电平台等用户触达渠道。

## 角色分工

| 角色        | 主要职责                       | 与调价平台的关系 |
| --------- | -------------------------- | -------- |
| 虚拟电厂      | 下发削峰、填谷调控指令，指定响应目标、时段和约束   | 指令来源     |
| 车网互动大脑    | 接收指令、识别事件类型、统筹执行侧和运营侧响应    | 调价平台的上游  |
| 调价平台      | 生成调价、优惠和活动建议，评估收益与响应能力     | 本方案主体    |

## 削峰场景逻辑

削峰时，虚拟电厂希望某些站点或区域在高峰时段减少充电负荷。

逻辑可以这样理解：

**虚拟电厂下发削峰指令**  
→ 车网互动大脑识别为削峰任务  
→ 车网互动大脑把任务传给调价平台  
→ 调价平台判断哪些站点适合参与削峰  
→ 根据以下因素制定价格策略：

- 当前站点预计负荷
- 站点历史削峰参与情况
- 历史调价效果
- 削峰带来的预计收益
- 价格调整对订单量、充电量、收入的影响

然后输出策略，例如：

- 高峰时段适当涨价
- 取消或降低优惠
- 引导用户去其他时段充电
- 引导用户去周边非削峰站点充电
- 对错峰充电用户发放优惠券

## 填谷场景逻辑

填谷时，虚拟电厂希望低谷时段增加充电负荷。

> 车网互动大脑不能直接让用户车辆增加充电功率，也不能强制用户来充电，所以需要调价平台通过价格和优惠引导用户在低谷时段来充电。

所以填谷逻辑是：

**虚拟电厂下发填谷指令**  
→ 车网互动大脑识别为填谷任务  
→ 车网互动大脑结合事件和站点历史数据，判断哪些站点有填谷潜力  
→ 调价平台根据站点价格敏感度、历史低谷充电情况，给出价格建议  
→ 车网互动大脑或调价平台计算可上报的填谷能力  
→ 上报虚拟电厂  
→ 同时把价格策略传导到易充平台 / 充电平台 / e充电  
→ 用户看到低价或优惠后，在低谷时段增加充电  
→ 最终形成填谷效果


## 方案设计

在现有智能调价外面加一层 **“车网互动任务调价层”**
现有仍然负责日常经营场景下的分时服务费优化；车网互动任务来了以后，不直接进入原调价流程 ，也不改变原来的训练逻辑，而是把现有输出的价格结果当作**基准经营价格**，即在现有最优价格基础上临时怎么调。

1.新增一个车网互动任务接口，如：

```
POST /vgi/optimize-pricing
```

它接收车网互动大脑下发的任务：

2. 现有 PPO 结果作为“**经营基准价**”

调用现有的功能，得到正常经营场景下的：

```
优化后的服务费
预期订单
预期收入
```

这个结果就是 **无车网互动任务时的基准策略**。

后续削峰、填谷都在这个基准价格上做叠加

3. 新增调价层，不进入原 reward
新增一个单独的任务评分函数：

```
VGI_score =  
任务完成度
+ 虚拟电厂补偿收益
+ 原经营收益保留
- 用户流失惩罚
- 排队风险惩罚
- 毛利损失惩罚
```

普通 PPO 的 reward 不变。
车网互动任务只在新增里使用自己的评分函数。

4. 选择任务完成度最高、收益损失最小的方案  
5. 输出任务型调价策略

## 削峰任务

削峰任务的目标是降低目标时段负荷。

算法上做法是：

```
基准负荷 - 调价后预测负荷 = 预计削峰量
```

对于削峰，只生成涨价或减少优惠的候选动作：

```
peak_shaving_actions = [0.00, +0.05, +0.10, +0.15, +0.20]
```

每个动作都在原 PPO 推荐服务费基础上叠加：

```
candidate_price = base_service_fee + delta
```

然后用你现有的车辆仿真逻辑模拟用户是否还来本站、是否换站、是否延后充电。

## 填谷任务

填谷任务的目标是增加目标时段负荷。

算法上做法是：

```
调价后预测负荷 - 基准负荷 = 预计填谷量
```

对于填谷，只生成降价或优惠动作：

```
valley_filling_actions = [0.00, -0.05, -0.10, -0.15, -0.20]
```

每个动作同样在现有 PPO 推荐价基础上叠加：

```
candidate_price = base_service_fee + delta
```

然后仿真车辆是否会因为低价转移到目标时段、目标站点充电。得到**填谷能力**
