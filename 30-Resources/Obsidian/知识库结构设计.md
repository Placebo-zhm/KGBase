# Obsidian 知识库结构设计

## 设计目标

这个知识库采用 `PARA + MOC` 的组织方式，用来支持个人学习、项目推进、资料沉淀和长期复用。

核心原则：

- 先收集，再整理，不要求每条笔记一开始就放到完美位置。
- 用项目推动行动，用领域维护长期责任，用资源沉淀参考材料。
- 用 MOC（Map of Content，内容地图）建立主题索引，避免只依赖目录层级。
- 旧内容不删除优先归档，保留上下文和演化过程。

## 推荐目录结构

```text
00-Inbox/
10-Projects/
20-Areas/
30-Resources/
40-Archives/
90-MOCs/
99-Attachments/
```

### 00-Inbox

临时收集区，用于放置还没有整理的想法、摘录、灵感、待处理材料。

适合放入：

- 临时想法
- 网页摘录
- 会议或课堂速记
- 还没有判断归属的笔记

使用规则：

- Inbox 不是长期存放区。
- 每周至少清理一次。
- 清理时将内容移动到项目、领域、资源或归档中。

### 10-Projects

项目区，用于放置有明确目标、阶段、完成状态的内容。

适合放入：

- 正在完成的课程任务
- 论文、实验、开发任务
- 有截止时间的学习计划
- 某个专题输出，如文章、报告、演示文稿

判断标准：

- 有明确结果。
- 有开始和结束。
- 可以被完成、暂停或归档。

示例：

```text
10-Projects/
  深度学习课程复习/
  Obsidian知识库搭建/
  电网可视化实验/
```

### 20-Areas

领域区，用于放置需要长期维护、没有明确结束时间的方向。

适合放入：

- 专业方向
- 长期技能
- 工作职责
- 健康、财务、写作等长期主题

判断标准：

- 没有明确完成日期。
- 需要持续维护。
- 会不断产生新的项目和资源。

示例：

```text
20-Areas/
  后端开发/
  人工智能/
  电力系统/
  英语学习/
```

### 30-Resources

资源区，用于存放主题资料、课程笔记、参考材料和可复用信息。

适合放入：

- 书籍笔记
- 课程资料
- 论文阅读
- 工具用法
- 技术文档摘要

判断标准：

- 当前不一定要执行。
- 以后可能会复用。
- 更偏参考和资料，而不是任务。

示例：

```text
30-Resources/
  Git/
  Obsidian/
  Python/
  LLM/
```

### 40-Archives

归档区，用于存放已完成、暂停、过期或暂时不用的内容。

适合放入：

- 完成的项目
- 过期计划
- 不再维护的资料
- 已经整理完的临时材料

使用规则：

- 归档不等于删除。
- 如果内容未来还可能参考，优先归档。
- 如果内容完全无价值，再考虑删除。

### 90-MOCs

MOC 区用于建立主题索引页。MOC 不一定保存完整内容，它主要负责组织链接、路径和学习脉络。

适合放入：

- 某个领域的导航页
- 某门课程的总索引
- 某个技术主题的知识地图
- 某个项目的入口页

MOC 示例：

```markdown
# Git MOC

## 基础

- [[Git 基本概念]]
- [[Git 常用命令]]

## Obsidian 同步

- [[Git管理指南]]
- [[Obsidian Git 冲突处理]]

## 待补充

- [ ] 整理 rebase 和 merge 的区别
- [ ] 补充远程仓库恢复流程
```

### 99-Attachments

附件区用于统一保存图片、PDF、压缩包和其他非 Markdown 文件。

推荐结构：

```text
99-Attachments/
  images/
  pdfs/
  files/
```

使用规则：

- 图片不要长期散落在根目录。
- Obsidian 设置中建议将附件默认保存位置改为 `99-Attachments/images`。
- 大文件谨慎加入 Git，必要时使用网盘或 Git LFS。

## 笔记类型

### 临时笔记

快速记录还没有整理的内容，默认进入 `00-Inbox`。

要求：

- 可以粗糙。
- 可以不完整。
- 后续必须整理、合并、移动或删除。

### 资料笔记

来自书籍、课程、论文、网页、视频的内容，默认进入 `30-Resources`。

建议包含：

- 来源
- 摘要
- 关键观点
- 自己的理解
- 后续可连接的主题

### 永久笔记

经过理解和重写的知识单元，应该能脱离原始资料独立阅读。

要求：

- 标题具体。
- 内容聚焦一个明确问题。
- 尽量使用自己的话表达。
- 主动链接到相关笔记或 MOC。

### 项目笔记

围绕一个目标推进的工作记录，默认进入 `10-Projects`。

建议包含：

- 目标
- 当前状态
- 待办事项
- 决策记录
- 复盘结论

### MOC 索引页

用于组织一个主题下的关键笔记和学习路径，默认进入 `90-MOCs`。

要求：

- 不追求收录所有笔记，只收录关键入口。
- 定期更新。
- 可以按基础、进阶、实践、问题、待整理等结构组织。

### 复盘笔记

用于总结项目、学习阶段或问题处理过程。

建议包含：

- 背景
- 做了什么
- 结果如何
- 有哪些问题
- 下次如何改进

## 命名规范

### 普通笔记

```text
主题 - 具体内容.md
```

示例：

```text
Git - 常用命令.md
Obsidian - 双链使用方法.md
深度学习 - 反向传播理解.md
```

### 项目笔记

```text
项目名 - 任务或阶段.md
```

示例：

```text
知识库搭建 - 目录设计.md
电网可视化实验 - 数据处理.md
```

### 日志和记录

```text
YYYY-MM-DD 内容.md
```

示例：

```text
2026-05-09 知识库整理记录.md
2026-05-09 Git同步问题处理.md
```

### 附件

附件统一放入 `99-Attachments`，不要长期散落在根目录。

推荐命名：

```text
YYYYMMDD-主题-说明.png
YYYYMMDD-主题-说明.pdf
```

示例：

```text
20260509-obsidian-graph-view.png
20260509-git-error-safe-directory.png
```

## 日常整理流程

### 快速记录

1. 新想法先进入 `00-Inbox`。
2. 不在记录时纠结分类。
3. 保留上下文，方便后续整理。

### 每日整理

1. 清理当天新增笔记。
2. 给重要笔记补充标题、链接和标签。
3. 将临时笔记移动到合适目录。
4. 对完成的整理执行 Git 提交。

### 每周整理

1. 清空或压缩 `00-Inbox`。
2. 更新重要 MOC。
3. 检查正在进行的项目。
4. 将完成或暂停的项目移动到 `40-Archives`。
5. 检查根目录是否有散落附件。

## 链接和标签规范

优先使用双链连接具体笔记：

```markdown
[[Git - 常用命令]]
[[Obsidian - 双链使用方法]]
```

标签用于状态或横向属性，不建议代替目录和 MOC。

推荐标签：

```text
#todo
#review
#idea
#source/book
#source/paper
#source/video
```

避免创建过多近义标签，例如同时使用 `#AI`、`#人工智能`、`#机器智能` 表示同一件事。

## 判断笔记应该放在哪里

可以用以下问题判断：

- 只是临时记录，还没整理？放入 `00-Inbox`。
- 有明确目标和完成状态？放入 `10-Projects`。
- 是长期维护的方向？放入 `20-Areas`。
- 是以后可能复用的资料？放入 `30-Resources`。
- 已完成、暂停或过期？放入 `40-Archives`。
- 是主题入口和导航？放入 `90-MOCs`。
- 是图片、PDF 或其他附件？放入 `99-Attachments`。

## 最小可执行规则

如果不想维护太复杂，只需要坚持以下规则：

1. 新内容先进入 `00-Inbox`。
2. 每周清理一次 Inbox。
3. 项目放 `10-Projects`，资料放 `30-Resources`，长期方向放 `20-Areas`。
4. 每个重要主题至少有一个 MOC。
5. 附件统一放 `99-Attachments`。
6. 每次完成整理后提交到 Git。
