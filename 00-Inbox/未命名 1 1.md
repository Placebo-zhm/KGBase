可以。第一步建议不要叫“从文件中抽取规则”，而是拆成两层：

> **文件 → 证据片段 Evidence Units → VMCT 候选规则**

也就是说，系统第一步不要直接生成最终规则集合，而是先从一个或多个文件中找到“可能支持规则的证据句/证据段”，再把这些证据转成结构化的 **变量—机制—条件—目标变量** 规则候选。

原 ExoLLM 的代码里是人工写死 `nature_attribute_template`、`trend_template`、`period_template` 等模板，再拼接成 prompts；你的第一步就是要把这些人工模板的来源换成外部文件中的证据。

---

## 总体流程

```text
文件上传
  ↓
文档解析
  ↓
文本切分
  ↓
变量/目标变量相关证据检索
  ↓
VMCT 结构化抽取
  ↓
证据校验与去噪
  ↓
候选规则池
```

核心输出不是最终 prompt，而是这种候选规则：

```json
{
  "variable": "T (degC)",
  "variable_alias": "air temperature",
  "mechanism": "photosynthetic activity",
  "condition": "daytime with sufficient radiation and active vegetation",
  "target": "CO2 (ppm)",
  "effect": "may reduce",
  "evidence_text": "During daytime, higher temperature can enhance photosynthetic activity when sufficient radiation and vegetation are present, which may reduce local CO2 concentration.",
  "source_file": "Weather_and_CO2_Dynamics.pdf",
  "page": 5,
  "confidence": 0.86
}
```

# 1. 从一个文件中怎么抽取

假设用户上传一个 PDF、DOCX 或 TXT 文件。流程如下。

## Step 1：文档解析

不同文件格式走不同 parser：

|文件类型|解析方式|
|---|---|
|PDF|提取正文、页码、标题、表格 caption|
|DOCX|提取段落、标题、表格|
|TXT / MD|直接按段落解析|
|CSV|一般不作为外部文本证据，而是用来获取变量名和数据上下文|
|HTML|提取正文、标题、来源链接|

解析后统一成这种结构：

```json
{
  "doc_id": "D001",
  "file_name": "Weather_and_CO2_Dynamics.pdf",
  "blocks": [
    {
      "block_id": "D001_B001",
      "page": 3,
      "section": "Carbon exchange and meteorological factors",
      "text": "Air temperature influences atmospheric CO2 concentration through photosynthesis and respiration..."
    }
  ]
}
```

这一步最重要的是保留 **来源信息**：文件名、页码、章节、段落编号。否则后面规则不可溯源。

---

## Step 2：文本切分

不要整篇文档丢给 LLM。应该切成较小的 evidence candidates。

推荐单位：

```text
1 个段落
或
1～3 个连续句子窗口
```

例如原文：

```text
Air temperature influences atmospheric CO2 concentration through two major biological processes. 
During daytime, higher temperature can enhance photosynthetic activity when sufficient radiation and vegetation are present, which may reduce local CO2 concentration. 
At night, higher temperature can increase plant and soil respiration, which may release more CO2 into the atmosphere.
```

切成：

```text
E1: Air temperature influences atmospheric CO2 concentration through two major biological processes.

E2: During daytime, higher temperature can enhance photosynthetic activity when sufficient radiation and vegetation are present, which may reduce local CO2 concentration.

E3: At night, higher temperature can increase plant and soil respiration, which may release more CO2 into the atmosphere.
```

每个证据片段都保留位置：

```json
{
  "evidence_id": "E2",
  "doc_id": "D001",
  "page": 3,
  "text": "During daytime..."
}
```

---

## Step 3：基于变量和目标变量筛选证据

这一步是为了避免对全文做昂贵抽取。

输入当前任务信息：

```json
{
  "target_variable": "CO2 (ppm)",
  "exogenous_variables": [
    "T (degC)",
    "rh (%)",
    "wv (m/s)",
    "rain (mm)",
    "PAR (mol/mol/s)"
  ]
}
```

先建立变量别名字典：

```json
{
  "T (degC)": ["temperature", "air temperature", "ambient temperature"],
  "rh (%)": ["relative humidity", "humidity"],
  "wv (m/s)": ["wind speed", "wind velocity"],
  "CO2 (ppm)": ["CO2", "carbon dioxide", "CO2 concentration", "atmospheric CO2 concentration"],
  "PAR": ["photosynthetically active radiation", "solar radiation", "light availability"]
}
```

然后筛选 evidence。

证据片段满足以下条件之一就保留：

```text
1. 同时包含变量别名和目标变量别名
2. 包含变量别名 + 机制触发词
3. 包含目标变量别名 + 机制触发词
4. 语义向量检索与 query 高相似
```

机制触发词可以包括：

```text
influence, affect, regulate, drive, modulate, increase, decrease,
through, by, due to, under, during, condition, mechanism, pathway,
photosynthesis, respiration, diffusion, dispersion, mixing, transport
```

例如：

```text
During daytime, higher temperature can enhance photosynthetic activity ...
```

会被保留，因为它包含：

```text
temperature + photosynthetic activity + condition daytime
```

---

## Step 4：VMCT 结构化抽取

对筛出来的证据片段，用 LLM 或信息抽取模型做 schema filling。

不要让模型自由总结，而是强制输出 JSON：

```json
{
  "variable": "",
  "mechanism": "",
  "condition": "",
  "target": "",
  "effect": "",
  "evidence_span": "",
  "certainty": "",
  "temporal_property": ""
}
```

例如输入证据句：

```text
During daytime, higher temperature can enhance photosynthetic activity when sufficient radiation and vegetation are present, which may reduce local CO2 concentration.
```

输出：

```json
{
  "variable": "temperature",
  "mechanism": "photosynthetic activity",
  "condition": "daytime with sufficient radiation and vegetation",
  "target": "CO2 concentration",
  "effect": "may reduce",
  "evidence_span": "During daytime, higher temperature can enhance photosynthetic activity when sufficient radiation and vegetation are present, which may reduce local CO2 concentration.",
  "certainty": "uncertain / conditional",
  "temporal_property": "daytime-dependent"
}
```

然后再做 entity linking：

```text
temperature → T (degC)
CO2 concentration → CO2 (ppm)
```

得到：

```json
{
  "variable": "T (degC)",
  "mechanism": "photosynthetic activity",
  "condition": "daytime with sufficient radiation and vegetation",
  "target": "CO2 (ppm)",
  "effect": "may reduce"
}
```

---

## Step 5：证据校验

这一阶段很关键，否则 LLM 容易补充不存在的机制或条件。

校验规则：

```text
1. evidence_span 必须能在原文中找到
2. variable 必须能映射到当前变量或其别名
3. target 必须能映射到当前目标变量或其别名
4. mechanism 不能是空泛词，比如 “influence” “relationship”
5. condition 如果原文没有，则允许为空，不强行生成
6. effect 如果原文没有方向，则标记为 “affect / uncertain”
```

例如：

```json
{
  "mechanism": "carbon cycle"
}
```

太泛，应该降权。

```json
{
  "mechanism": "photosynthetic carbon uptake"
}
```

更具体，应该保留。

# 2. 从多个文件中怎么获取

多个文件不是简单拼接，而是走 **文档级抽取 + 全局融合**。

流程是：

```text
File 1 → Evidence Units → VMCT Candidates
File 2 → Evidence Units → VMCT Candidates
File 3 → Evidence Units → VMCT Candidates
                    ↓
          Candidate Rule Pool
                    ↓
        Entity Normalization
                    ↓
        Rule Fusion / Conflict Handling
                    ↓
              Rule Knowledge Base
```

---

## 多文件阶段的关键问题

### 1. 同义实体合并

不同文件可能这样写：

```text
air temperature
ambient temperature
temperature
T
```

都要合并到：

```text
T (degC)
```

不同文件可能这样写：

```text
carbon dioxide concentration
atmospheric CO2
CO2 level
CO2 concentration
```

都要合并到：

```text
CO2 (ppm)
```

所以多文件处理必须有：

```text
Entity Resolution / Variable Linking
```

---

### 2. 相似机制合并

不同文件可能说：

```text
photosynthesis
photosynthetic activity
photosynthetic carbon uptake
carbon assimilation
plant carbon fixation
```

这些不应该变成 5 条完全重复规则，而是聚成一个机制簇：

```text
photosynthetic carbon uptake
```

最终规则可以保留多条 evidence：

```json
{
  "variable": "PAR",
  "mechanism": "photosynthetic carbon uptake",
  "condition": "daytime and vegetation-active condition",
  "target": "CO2 (ppm)",
  "effect": "may reduce",
  "evidence_list": [
    "D001_E12",
    "D003_E07",
    "D005_E21"
  ],
  "support_count": 3
}
```

---

### 3. 冲突不是删除，而是条件化

多个文件可能出现看似冲突的规则。

例如：

```text
temperature increases CO2 through respiration.
temperature decreases CO2 through photosynthesis.
```

不能简单合并成：

```text
temperature affects CO2
```

也不能删掉其中一个。

应该保留为两条条件化规则：

```json
[
  {
    "variable": "T (degC)",
    "mechanism": "plant and soil respiration",
    "condition": "nighttime or respiration-dominant conditions",
    "target": "CO2 (ppm)",
    "effect": "may increase"
  },
  {
    "variable": "T (degC)",
    "mechanism": "photosynthetic carbon uptake",
    "condition": "daytime with sufficient radiation and vegetation",
    "target": "CO2 (ppm)",
    "effect": "may reduce"
  }
]
```

这就是你设计 `condition` 字段的意义。

# 3. 一个推荐的实现架构

我建议你第一步实现成 4 个模块：

```text
Document Parser
Evidence Retriever
VMCT Extractor
Rule Candidate Manager
```

## Module 1：Document Parser

输入：

```text
PDF / DOCX / TXT / MD / HTML
```

输出：

```json
[
  {
    "doc_id": "D001",
    "block_id": "B001",
    "page": 3,
    "section": "xxx",
    "text": "..."
  }
]
```

## Module 2：Evidence Retriever

输入：

```json
{
  "blocks": [...],
  "variables": ["T (degC)", "rh (%)", "wv (m/s)"],
  "target": "CO2 (ppm)"
}
```

输出：

```json
[
  {
    "evidence_id": "E001",
    "text": "...",
    "matched_variable": "T (degC)",
    "matched_target": "CO2 (ppm)",
    "retrieval_reason": "variable-target-mechanism co-occurrence"
  }
]
```

## Module 3：VMCT Extractor

输入 evidence sentence。

输出：

```json
{
  "variable": "T (degC)",
  "mechanism": "photosynthetic activity",
  "condition": "daytime with sufficient radiation",
  "target": "CO2 (ppm)",
  "effect": "may reduce",
  "evidence_id": "E001"
}
```

## Module 4：Rule Candidate Manager

负责：

```text
去重
合并
实体对齐
证据聚合
置信度计算
冲突标记
```

输出候选规则池：

```json
{
  "candidate_rules": [...],
  "evidence_table": [...],
  "entity_mapping": {...}
}
```

# 4. 单文件和多文件的区别

|项目|单文件|多文件|
|---|---|---|
|抽取方式|文档内 evidence → rule|每个文件独立抽取|
|主要难点|找准证据句|跨文件合并、去重、冲突|
|输出|一个候选规则列表|全局候选规则池|
|置信度|主要看 evidence quality|还要看 cross-source support|
|适合场景|demo、MVP|正式知识库构建|

单文件可以先做通；多文件只是在它上面加一层：

```text
跨文档 entity resolution + rule fusion + evidence aggregation
```


# 6. 用一句话总结

第一步不要直接从文件生成 prompt，而是：

> **从一个或多个文件中解析文本，围绕当前变量和目标变量检索证据片段，再用 VMCT schema 将证据片段结构化为候选规则；多个文件的结果通过实体对齐、机制聚类、证据聚合和冲突条件化形成候选规则池。**

这个候选规则池就是后续图谱构建和规则集合生成的基础。
